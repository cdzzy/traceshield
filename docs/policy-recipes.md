# traceshield 策略落地与合规报告手册

> 对齐今日 Trending 项目中“运行时护栏”“审计可追溯”“面向企业交付的合规输出”三类最佳实践。

## 为什么策略系统不能只停留在拦截

一个可用的 Agent 安全层，至少要同时回答三件事：

1. 这个动作是否应该被允许
2. 如果被拦截，为什么拦截
3. 事后如何生成可审计、可交付的报告

如果项目已经加入 SOC2 / GDPR 报告生成器，下一步就应该把“策略执行”和“报告产出”串成一条完整链路。

## 推荐的策略执行链路

### 1. Ingest

接收事件时统一补全：

- actor
- action
- target
- data_classification
- trace_id
- timestamp

### 2. Evaluate

建议策略规则至少覆盖：

- 数据访问边界
- 工具调用白名单
- 外发内容限制
- 敏感词 / 敏感字段
- 匿名化要求

### 3. Decide

决策不要只有 allow / deny，建议扩展到：

- allow
- allow_with_redaction
- require_approval
- deny

### 4. Record

把决策写入不可篡改审计轨迹，并保留：

- 触发规则
- 输入摘要
- 决策结果
- 处理耗时
- 关联 trace_id

### 5. Report

在 SOC2 / GDPR 报告里，不要只列总数，建议额外输出：

- 高风险事件分布
- 典型拦截案例
- 数据分类命中情况
- 需要人工复核的条目

## 推荐的规则分层

### 基础层

- PII 检测
- 禁止明文密钥外发
- 外部请求域名白名单

### 运行时层

- 高风险工具二次确认
- 大批量导出限流
- 敏感上下文脱敏

### 合规层

- GDPR 删除请求可追踪
- 审计日志保留期
- 合规报告导出频率

## 事故复盘建议

每次高风险拦截后，建议固定输出：

- 事件摘要
- 命中策略
- 是否误报
- 后续修复动作
- 是否需要新增规则
